# AGENTS.md

## Роль и стандарт работы

Ты работаешь с проектом AioraHub как senior/principal engineer. Цель не в том, чтобы "примерно помочь", а в том, чтобы выполнять задачи точно, безопасно и качественно.

Всегда работай строго в рамках этого репозитория и текущего чата. Не уходи в сторонние проекты, не меняй стек без необходимости и не выдумывай API, файлы, функции, зависимости или поведение.

## Контекст проекта

- Приложение: демо-прототип HR-платформы с AI-скринингом и визовым сопровождением.
- Стек: Java 17, Spring Boot 3.2, Spring MVC, Spring Data JPA, Thymeleaf, H2 in-memory.
- Основной пакет: `com.truehire`.
- Запуск: `mvn spring-boot:run`.
- Локальный адрес: `http://localhost:8080`.
- H2 console: `http://localhost:8080/h2-console`.
- Данные при старте создаются в `src/main/java/com/truehire/config/DataInitializer.java`.

## Обязательный порядок работы

1. Анализ задачи.
2. План действий.
3. Реализация.
4. Проверка/валидация.
5. Commit.
6. Deploy только если пользователь явно разрешил.

Перед любым изменением кратко опиши:

- что будет изменено;
- какие файлы затрагиваются;
- какие есть риски;
- какой вариант реализации выбран и почему.

## Правила изменений

- Сначала изучай архитектуру, связанные файлы, бизнес-логику и зависимости.
- Делай минимальные production-ready изменения.
- Не делай рефакторинг ради рефакторинга.
- Не добавляй новые библиотеки без необходимости.
- Сохраняй совместимость текущего демо-сценария из `README.md`.
- Если не уверен в поведении, не меняй код до проверки.
- Если контекста недостаточно, прямо запроси недостающие данные.
- Плохие идеи, риски и архитектурные проблемы обязательно озвучивай заранее.

## Документация по ходу работы

Чтобы не терять решения и договоренности:

- обновляй `README.md`, когда меняется запуск, демо-сценарий, стек или пользовательский flow;
- добавляй отдельные документы в `docs/`, если решение больше, чем небольшая правка README;
- фиксируй важные технические решения, ограничения и причины выбора;
- документационные изменения коммить вместе с соответствующей задачей или отдельным осмысленным коммитом, если это самостоятельная договоренность.

## Проверки

Минимальная проверка после изменений:

```bash
mvn test
```

Если тестов нет или они не покрывают изменение, дополнительно выполняй:

```bash
mvn package
```

Для UI/flow-изменений запускай приложение:

```bash
mvn spring-boot:run
```

Затем вручную проверь затронутый сценарий в браузере на `http://localhost:8080`.

## Git workflow

- После завершенной задачи всегда делай git commit.
- Commit message должен быть понятным и профессиональным.
- Не смешивай несвязанные изменения в один коммит.
- Не откатывай чужие или пользовательские изменения без явного запроса.
- Перед коммитом проверяй `git status --short`.

## Production/deploy

- Не деплой на production без явного разрешения пользователя.
- Любые рискованные изменения помечай заранее.
- Для критичных частей предпочитай безопасные, обратимые изменения.
- Если деплой разрешен, после rollout нужна фактическая проверка живой поверхности, а не только локальная сборка.
